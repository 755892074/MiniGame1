# 04 游戏框架调研报告

---

## 1. 候选框架

| 框架 | 仓库 | Stars | 定位 |
|------|------|-------|------|
| F8Framework | github.com/TippingGame/F8Framework | 3.2k+ | 全栈工具箱型 |
| QFramework | github.com/liangxiegame/QFramework | 4.5k+ | 架构方法论型 |

## 2. 详细对比

### 2.1 F8Framework

**核心理念**："F8 一键启动"，所有模块通过 `F8.Run()` 一行代码初始化。

| 维度 | 评价 |
|------|------|
| 上手曲线 | ⭐⭐⭐⭐⭐ 极低，Demo 复制即用 |
| 内置模块 | 19 个：UI/音频/事件/对象池/FSM/输入/资源/设置/本地化/网络/SDK管理/计时器/协程/寻路/Tween/物理/热更/数据/场景 |
| 抖音支持 | ⭐⭐⭐⭐⭐ SDK 管理模块原生支持 |
| 文档质量 | ⭐⭐⭐⭐ 中文文档 + B站视频 + Demo 项目 |
| 社区活跃 | ⭐⭐⭐ 中等，核心维护者活跃 |
| 适合规模 | 中小型项目（< 50 个脚本） |
| 扩展性 | 模块化设计，按需加载 |
| 劣势 | 社区相对小，问题响应不如大框架 |

### 2.2 QFramework

**核心理念**：架构优先，提供 MVC/CQRS/DDD/IOC/命令/事件等多种架构模式。

| 维度 | 评价 |
|------|------|
| 上手曲线 | ⭐⭐⭐ 中等，需理解架构概念 |
| 内置模块 | 架构模式为主，工具模块较少 |
| 抖音支持 | ⭐⭐ 需自行适配 |
| 文档质量 | ⭐⭐⭐⭐⭐ 完整中英文文档 + 视频教程 |
| 社区活跃 | ⭐⭐⭐⭐ QQ群活跃，作者本人回答 |
| 适合规模 | 中大型项目（需要架构约束） |
| 扩展性 | 架构层面强，工具层面弱 |
| 劣势 | 架构概念学习成本高，过度设计风险 |

### 2.3 GameFramework

**核心理念**：商业项目标准框架，参考 Unity 官方最佳实践。

| 维度 | 评价 |
|------|------|
| 上手曲线 | ⭐⭐ 陡峭，模块多、配置复杂 |
| 内置模块 | 数据表/实体/事件/有限状态机/本地化/网络/对象池/流程/资源/场景/配置/UI/声音/Web请求 |
| 抖音支持 | ⭐ 无原生支持 |
| 文档质量 | ⭐⭐⭐ 完整但偏技术 |
| 社区活跃 | ⭐⭐⭐ GitHub 活跃 |
| 适合规模 | 中大型商业项目 |
| 扩展性 | 框架层面强，但不够灵活 |
| 劣势 | 太重，对 2D 小游戏过度设计 |

### 2.4 自研轻框架

**核心理念**：只写需要的部分。

| 维度 | 评价 |
|------|------|
| 上手曲线 | ⭐⭐⭐⭐⭐ 最了解自己写的东西 |
| 内置模块 | 按需开发 |
| 抖音支持 | ⭐⭐⭐ 需自行对接 |
| 文档质量 | ⭐⭐ 无现成文档 |
| 社区活跃 | ⭐ 无 |
| 适合规模 | 原型验证 |
| 劣势 | 重复造轮子，维护成本高 |

## 3. 最终选择：F8Framework

### 3.1 选择理由

```
匹配度分析（本项目需求 vs 框架能力）：

需求                      F8Framework    QFramework    自研
─────────────────────────────────────────────────────────
快速原型              ✅ 一键启动     ❌ 需学架构    ✅ 灵活
2D 小游戏             ✅ 内置2D模块   ✅             ✅
抖音平台              ✅ 原生支持     ❌ 需适配     ❌ 需写
广告SDK管理           ✅ SDK模块      ❌ 需自己写   ❌ 需写
对象池（UI重用）       ✅ 内置        ⚠️ 需自己写    ❌ 需写
音频管理              ✅ 内置        ⚠️ 简单        ❌ 需写
有限状态机            ✅ 内置        ✅ 有          ❌ 需写
轻量级                ✅ 按需加载     ⚠️ 架构重     ✅
单人开发友好           ✅ 极低门槛     ⚠️ 学习成本   ✅

结论：F8Framework = 8 项 ✅ vs QFramework = 2 项 ✅
```

### 3.2 风险缓解

| 风险 | 缓解措施 |
|------|---------|
| 框架作者断更 | F8Framework 已稳定迭代3年，核心模块已成熟 |
| 社区小 | 核心功能足够，不需重度依赖社区 |
| 版本兼容 | 固定框架版本号，不随意升级 |

### 3.3 使用原则

- **只加载需要的模块**，不用的不初始化
- **不过度封装**，F8 模块足够用就不二次封装
- **保持框架版本锁**，在 manifest.json 中锁定版本
- **遇到框架Bug先看源码**，F8 源码清晰可读
