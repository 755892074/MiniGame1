## 2. 分支策略

```
main ─────●────●────────────●────  (受保护)
           \                /
develop ───●──●──●────●──●──────  (开发主线)
              \    \  /
               \    release/v1.0.0
                \
        feature/game1-finddiff
```

### 2.1 分支命名规则

| 分支类型 | 命名 | 示例 |
|----------|------|------|
| 功能开发 | `feature/{描述}` | `feature/level-select-ui` |
| 游戏1 | `feature/game1-{描述}` | `feature/game1-diff-detect` |
| 游戏2 | `feature/game2-{描述}` | `feature/game2-combo-system` |
| Bug修复 | `fix/{描述}` | `fix/ad-loading-crash` |
| 发布 | `release/v{版本}` | `release/v1.0.0` |
| 热修复 | `hotfix/{描述}` | `hotfix/critical-ad-bug` |

### 2.2 分支生命周期

1. `feature/*` 从 `develop` 创建 → 开发 → PR → 合并到 `develop` → 删除
2. `release/*` 从 `develop` 创建 → 修复Bug → 合并到 `main` + `develop` → 删除
3. `hotfix/*` 从 `main` 创建 → 修复 → 合并到 `main` + `develop` → 删除

## 3. 提交规范

### 3.1 Conventional Commits

```
<type>[(scope)]: <描述>

type: feat / fix / docs / refactor / test / chore / perf / style
scope: 可选，模块名或游戏名

示例：
feat(finddiff): 添加差异点检测核心逻辑
feat(quiz): 实现连击计分系统
fix: 修复广告SDK初始化时序问题
docs: 更新企划书中的竞品分析
refactor(core): 提取 DataStorage 为独立模块
test: 添加 DifferenceManager 单元测试
chore: 配置 Git LFS
perf: 优化关卡图加载，使用异步加载
```

### 3.2 提交粒度

- 一个提交 = 一个逻辑变更
- 不要在一个提交中混合"新功能"和"重构"
- 提交信息用中文或英文均可，保持统一

### 3.3 多人协作（未来）

如果以后有其他开发者加入：
- 所有代码通过 PR 合并
- PR 标题遵循同样的 Commit 格式
- 至少自己 review 一遍再提 PR

## 4. 版本号管理

### 4.1 语义化版本

```
v{主版本}.{次版本}.{修订号}

主版本 (MAJOR): 重大架构变更、不兼容的 API 改动
次版本 (MINOR): 新功能、新关卡、新章节
修订号 (PATCH): Bug 修复、小优化

示例：
v1.0.0 → 首次发布（10关找茬）
v1.1.0 → 新增10关
v1.1.1 → 修复一个Bug
v1.2.0 → 新增每日挑战功能
v2.0.0 → 新增无厘头大挑战
```

### 4.2 代码中版本号

```csharp
// 在 GameManager 中设置
public const string Version = "1.0.0";

void Start() {
    Debug.Log($"Game Version: {Version}");
    // 可在设置界面展示给玩家
}
```

### 4.3 发布流程

```
1. 确认 develop 分支稳定
2. 创建 release/vX.Y.Z 分支
3. 更新版本号（代码 + 文档）
4. 打包测试
5. 合并到 main → 打 Tag → 上传抖音
6. 合并回 develop
```
